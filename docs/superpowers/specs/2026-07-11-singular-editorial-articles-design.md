# AI Chronicle 非模板化文章体系设计

## 定位变化

AI Chronicle 不再把事件详情页当作知识卡片的展开页。时间轴仍负责准确、简洁地标记事件；详情页则成为一部关于技术、人物、机构、野心、误判和时代转向的文学编年史。

项目首先追求阅读本身的趣味。读者不必带着“解决一个问题”的目的进入文章，也可以因为一个人物、一个瞬间或一种历史气氛继续读下去。

文学性不能建立在虚构上。日期、数字、论文结论、比赛过程、人物言论、产品能力和不确定性必须能回到来源。没有来源支持的对话、心理活动、场景细节和因果关系不得补写。

## 只统一底线，不统一文章

所有文章共享以下编辑底线：

- 事实可以核对，判断与事实不混写。
- 引用不改写成作者自己的话，无法找到原文就不使用引号。
- 技术细节必须准确，但只在叙事真正需要时进入正文。
- 不用“颠覆、划时代、历史铭记、彻底改变”等词代替论证。
- 不把每个模型发布都写成人类命运的转折。
- 不因追求文学气息而编造天气、神态、沉默、房间陈设或人物内心。
- 中文是原生文章，英文是基于同一事实账本重新写作，不逐句机械翻译。

除此之外，不规定文章结构。

尤其禁止建立以下隐形模板：

- 统一用一个戏剧场景开头。
- 固定按“时代困境—决定性时刻—后来影响”展开。
- 每篇都用人物命运承载技术史。
- 强制三到五个章节、统一字数或统一段落节奏。
- 在结尾重复“它改变了世界”或把事件强行抬升到今天。
- 为所有文章设置相同的小标题，例如“风暴之前”“命运时刻”“余波”。

有些文章可以没有人物，没有场景，没有小标题，甚至没有传统意义上的高潮。它们只需要找到属于该事件自己的写法。

## 每篇文章的独立写作决定

动笔前，作者必须先在工作记录中回答五个问题。这些回答用于编辑，不作为网页模板展示：

1. 这篇文章真正值得写的对象是什么？
2. 它适合怎样的文体、叙述距离和时间尺度？
3. 情感从哪些真实材料中自然产生？
4. 哪些常见叙述已经被写滥，这篇必须避开什么？
5. 遮住标题以后，读者能否仅凭文章的声音和材料认出这个事件？

写作形式可以包括但不限于：思想随笔、人物侧写、群像、现场报道、论文近读、工业史、制度史、产品史、失败史、社区史、短篇纪事或带有作者判断的历史评论。

形式由材料决定，不由事件等级决定。例如：

- 图灵测试可以是安静的思想随笔。
- 达特茅斯可以是带有反讽距离的群像。
- 第一次 AI 寒冬可以写承诺、预算和实验室如何一同冷却。
- Transformer 可以贴近论文语言与架构本身，不必虚构一个英雄时刻。
- AlphaGo 可以沿五局比赛的时间推进。
- ChatGPT 可以写一个没有发布会的产品如何进入日常生活。
- Stable Diffusion 可以写开放权重释放出的社区创造力与失控。
- 欧盟 AI 法案可以保留制度文本的缓慢和重量。

这些例子只说明差异，不是文章分类表。后续文章不得为了归类而模仿例子的结构。

## 内容模型

事件元数据与文章正文分离。

```text
content/events/<slug>.md
  日期、标题、摘要、人物、组织、主题、来源、关系

content/event-articles/zh/<slug>.md
  eventId + 自由 Markdown 中文正文

content/event-articles/en/<slug>.md
  eventId + 自由 Markdown 英文正文
```

事件元数据继续服务时间轴、搜索、筛选、SEO 和关系页。文章文件只承担阅读内容，不重复维护日期、公司和来源。

构建时按 `eventId` 连接三份内容。正式构建必须满足：

- 70 个已发布事件都有中文文章。
- 70 个已发布事件都有英文文章。
- 中英文文章的 `eventId` 唯一且能找到对应事件。
- 没有孤立文章或重复正文。

迁移完成后，详情页不再回退显示 `background`、`whatHappened`、`whyImportant`、`impact` 和 `beforeAfter`。这些旧字段保留一段时间作为事实账本，待文章核验完成后再决定是否清理。

## 文章页面

详情页保留现有设计系统和明暗主题，但改变阅读秩序。

页面由以下部分组成：

1. 标题和一句克制的副标题。
2. 位于正文之前的元数据卡片。
3. 连续的文章正文。
4. 文末来源与延伸阅读。
5. 相关事件和前后时间节点。

元数据卡片是唯一允许集中结构化信息的区域。它显示日期、重要程度、人物、组织、主题、模型、来源数量和编辑状态，并在最后设置一行横跨整张卡片的“史实”。史实用一到三句话客观交代事件主体、公开动作、直接结果与必要的时间边界，不解释历史意义，也不出现“核心变化”“为什么重要”等结论。移动端按两列收拢常规字段，史实仍独占整行；卡片结束后进入不被打断的文章正文。

史实直接取自事件元数据中的 `summary` / `summaryEn`，保持单一事实来源，不在中英文文章文件中重复维护。中文与英文各使用自己的原生摘要；缺少英文摘要时构建失败，不能回退为中文或依赖页面字典翻译。史实是元数据卡片的收束，也是正文阅读的事实入口，但不另起一张卡片、不添加第二套标题层级，也不与文章首段合并。

删除以下正文模块：

- 核心变化。
- 发生了什么。
- 当时的行业背景。
- 为什么重要。
- 在它之前 / 在它之后。
- 它改变了什么。

日期、主题、相关人物、组织和编辑状态集中在开篇元数据卡片中，不切断正文。来源保留明确编号和可访问链接。

排版目标：

- 中文正文阅读宽度约 680–720px。
- 桌面正文约 17px，行高 1.9–2；英文按实际字形微调。
- 段落之间有呼吸，但不把每句话拆成社交媒体短句。
- 小标题、引文、脚注和分隔只在文章本身需要时出现。
- 不添加装饰性首字下沉、假引语、通用时间轴插图或重复信息卡片。

## 内容深度

不按等级规定硬字数。长度服从材料与叙事：

- 一个复杂的 S 级事件可能需要长篇，也可能适合紧凑文章。
- 一个 B 级事件如果拥有独特人物和材料，可以比 A 级事件更长。
- 发布信息有限的小版本不靠背景百科凑篇幅。
- 写完后以“是否形成完整阅读经验”判断充分性，不以字数判断。

高质量文章至少满足三项：

- 有一个无法被其他事件替换的叙事中心。
- 出现具体且可核对的材料，而非只给结论。
- 技术与人的选择、组织条件或时代环境发生真实联系。
- 文章允许复杂态度存在，不把历史收束成单一教训。
- 结尾完成当前文章的情绪或思想运动，而非重复摘要。

## 图片与组织 Logo

事件文章采用“史料优先、原创补位”的媒体策略。每个事件至少拥有一个有叙事或证据价值的主视觉，不能用无关图库照片、通用 AI 光效或仅为填空而生成的装饰图。主视觉位于元数据卡片之后、文章正文之前，属于正文阅读流的一部分；只有当第二张图片提供不同证据时，才允许在正文中追加图片。

图片来源按以下顺序选择：

1. 可明确署名和追溯的历史档案、公共领域或 Creative Commons 图片。
2. 论文中的关键图、官方产品界面、发布材料与机构档案。
3. 无法获得可靠素材时，由 AI Chronicle 生成的原创编辑插图。

所有图片必须本地化保存，不依赖第三方热链。每张图片必须记录中英文替代文本、中英文图注、署名、原始来源、权利类型与原始尺寸；生成图必须保留生成提示词和生成方式，并在图注明确写作“AI Chronicle 原创插图 / AI Chronicle original illustration”。来源、版权或主体身份不明确的图片不得上线。

事件主视觉在事件元数据中维护，避免中英文文章重复引用：

```yaml
visual:
  src: /assets/events/<event-id>.webp
  width: 1600
  height: 900
  alt: 中文替代文本
  altEn: English alternative text
  caption: 中文图注
  captionEn: English caption
  credit: 作者或机构
  sourceUrl: https://...
  rights: public-domain | cc | official-editorial | generated
  prompt: 仅生成图需要
```

组织 Logo 由公司内容对象统一维护，并在事件元数据的“组织”、公司详情页和模型谱系中复用。Logo 只使用机构官方品牌资源或可核验的可信矢量库，不使用 Image Gen 模仿商标；缺少可靠 Logo 时保留公司文字名称。Logo 资产同样本地化，并记录来源。Logo 不计入事件主视觉数量。

媒体验收必须覆盖：资源存在、尺寸与格式有效、无明显压缩瑕疵、替代文本与图注齐全、来源可访问、权利类型合法、明暗主题下可读、移动端无裁切和横向溢出。

## 全量内容增补与去模板

现有文章不是最终底稿。每篇文章重新建立事实账本，至少核对两类可靠来源；如果某事件只有一份足够权威的一手材料，则在编辑记录中说明，不为凑数量增加弱来源。扩写必须增加事件独有的数字、原话边界、实验条件、产品机制、制度过程或后来才可确认的结果，而不是补背景百科和意义判断。

中文与英文分别进行深改。去模板检查除首尾相似、小标题和高频意象外，还必须人工复核否定对照句、意义总结句、统一高潮、同构段落、假金句和“不是 A 而是 B / not X but Y”式辩护。英文继续基于同一事实账本重新写作，不把扩写后的中文逐句翻译过去。

## 并行子代理写作

最多同时运行三个写作子代理，与主代理共同工作。每个子代理负责不重叠的事件集合，避免文件冲突。

### 第一轮：中文原稿

- 子代理 A：早期 AI、神经网络与基础设施。
- 子代理 B：深度学习、语言模型、生成模型与开放模型。
- 子代理 C：AI Coding、Agent、产品、治理与 AI for Science。
- 主代理：实现内容模型与页面，并亲自写关键校准文章；同时审阅各代理的独立写作决定。

分组只用于协调文件所有权，不规定组内文章的文体。

### 第二轮：交叉审稿

每个子代理审阅另一个代理的文章，重点寻找：

- 相同的开场句法。
- 重复的章节组织。
- 反复出现的比喻和“历史转折”措辞。
- 统一的高潮与收尾方式。
- 把技术史强行人物化或戏剧化。
- 事实账本遗漏与来源边界越界。

发现问题后直接修改，不只提交评价。

### 第三轮：英文重写

英文文章由没有写中文原稿的代理负责。作者先读取事实账本和完整中文文章，再按英文文学非虚构的节奏重新组织。英文必须保留事实和情感方向，但不复制中文句序、修辞和章节边界。

### 主代理终审

主代理负责：

- 所有 S 级事件与关键 A 级事件终审。
- 随机抽取每个子代理至少五篇做逐段核对。
- 检查 70 篇文章之间的结构、开头、结尾和高频修辞重复。
- 对事实有疑问的段落回查原始来源。
- 判断文章是否仍然像批量生成内容；不合格就退回重写。

## 去套路验证

构建检查之外增加编辑检查：

- 扫描文章首段和末段的高相似 n-gram。
- 统计跨文章重复小标题。
- 扫描高频空洞判断与固定转折句。
- 列出重复比喻、历史拟人和群星/火焰/风暴等意象，人工判断是否滥用。
- 连续阅读同一时期至少五篇，检查是否产生可预测节奏。

自动检查只负责找到可疑重复，不用分数判断文学质量。最终仍由逐篇阅读决定。

## 事实与来源

每篇文章开始前建立事实账本，至少包括：

- 时间、地点、人物、组织与公开动作。
- 论文或模型的具体贡献与限制。
- 数字、比分、参数、评测和成本的原始口径。
- 当时已知的信息与后来才知道的信息。
- 争议项、厂商自报项与尚未独立复现项。

来源优先顺序为论文、官方档案、当事人记录、监管原文和可靠一手报道。文学化表达不能提高事实的确定性。

## 验收标准

- 70 个事件详情页全部改为自由文章正文，不再出现固定问答模块。
- 每页正文前有一个克制的元数据卡片，卡片不承担叙事和结论。
- 中文和英文各 70 篇文章均可独立阅读。
- 每个事件的元数据卡片最后都有一行全宽史实，中英文均为原生内容，且只陈述可核验事实。
- 70 个事件均有至少一个本地化主视觉，图片、图注、替代文本、署名、来源和权利类型完整。
- 所有已收录组织均有可靠来源的本地 Logo，或明确回退为文字名称；不得出现生成或近似商标。
- 140 篇中英文正文完成事实增补、去模板复核和交叉审稿，不以增加字数代替增加材料。
- 任意连续阅读五篇，开场、章节、节奏和结尾不能呈现同一模板。
- 每篇文章都有独立写作决定和事实账本，并通过交叉审稿。
- 页面在桌面和移动端保持舒适阅读宽度，无正文卡片化和横向溢出。
- 明暗主题、中英文切换、搜索、筛选、来源和相关事件正常。
- 正式构建、站内链接与内容完整性检查通过。
- 提交推送到独立 GitHub 仓库，并部署到独立 Cloudflare Pages 项目；Worker 路由无变化时不重新部署。
