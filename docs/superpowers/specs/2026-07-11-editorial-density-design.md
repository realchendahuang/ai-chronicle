# AI Chronicle 编辑密度与内容扩充设计

## 目标

AI Chronicle 的首页仍然首先是一条时间轴。视觉层级退后一步，让日期、事件和事实成为主角；详情页则从简短百科升级为有来源、能解释技术变化与行业后果的编辑型档案。

这轮改造同时解决三个问题：标题过大、文案空泛、事件覆盖不足。项目继续维持独立仓库、独立构建与独立 Cloudflare Pages 部署，主域名仅通过既有 Worker 路由挂载。

## 信息架构

不增加新的一级导航，也不把首页改成门户或卡片墙。

- 首页：时间总览、筛选工具、按年份排列的事件流。
- 事件详情：日期与标题、事实摘要、背景、发生了什么、为何重要、影响、前后变化、相关事件、来源。
- 模型谱系：保留代际浏览，补齐各代定位、能力变化和发布时间。
- 概念与公司页：保留作为交叉索引，不与事件详情重复大段内容。

## 视觉与字号

沿用现有明暗主题、字体、颜色、时间轴结构和交互。此次是设计系统内的密度校正，不重新设计页面，也不引入图片资产。

- 首页主标题：桌面最大 62px，移动端最大 46px。
- 普通索引页标题：桌面最大 72px，移动端 44–48px。
- 事件详情标题：桌面最大 72px，移动端 42–46px。
- 概念、公司、主题和模型谱系标题：桌面最大 80px，不再出现 100px 以上标题。
- 正文：桌面 16–18px，移动端 16px；摘要最多 19px。
- 列表事件标题：桌面 20–24px，移动端 17–18px。
- 保持正文每行约 58–76 个中文字符的阅读宽度。

时间轴继续占据首页主画面。缩小标题后释放的空间用于显示更多年份与事件，而不是加入新的口号、统计卡片或装饰。

## 中文编辑规则

采用深度改写，但保护日期、数字、术语、产品名、论文名、引用和来源。

- 删除“彻底改变”“意义不只是”“奠定基础”“引发热潮”等没有事实支撑的定性。
- 摘要第一句直接说明谁在何时发布或完成了什么。
- “为何重要”必须指出可观察的技术、产品或行业变化，不能重复摘要。
- “影响”按真实受影响对象填写，不强求开发者、用户、产品、行业四栏齐全。
- 不为句式整齐而写三段排比，不重复解释同一个结论。
- 没有材料支撑的判断直接删除，不用近义词包装。
- 中英文语义一致，但英文按英文信息写作习惯重写，不逐字翻译中文。

## 单个事件的内容标准

重点事件应尽量回答以下问题：

1. 当时已有怎样的技术或市场背景？
2. 具体发布了什么，架构、规模、产品形态或使用方式发生了什么变化？
3. 有哪些关键数字、论文结果或官方定义可以核对？
4. 它解决了什么旧问题，又留下了什么限制？
5. 对研究者、开发者、用户或行业的直接影响是什么？
6. 后来哪些事件继承、修正或放大了它？

不是每个事件都机械填满六项。信息不足时保持短，避免用套话凑长度。

## 扩充范围

第一轮目标为 60–80 个已发布事件，优先补齐断层，而不是平均撒点。

- 基础研究：图灵测试、感知机、专家系统、反向传播、ImageNet、AlexNet、word2vec、seq2seq、Transformer、BERT、扩散模型、AlphaFold。
- 大模型：GPT、Claude、Gemini、Llama、DeepSeek 各代关键节点，以及训练、对齐、上下文和推理能力变化。
- 产品与开发工具：ChatGPT、Copilot、Claude Code、MCP、工具调用、AI Agent 与主要编码 Agent 节点。
- 开源与基础设施：CUDA、开放权重模型、MoE、关键算力与推理成本节点。
- 社会与治理：有明确行业后果的监管、版权、安全与公司治理事件。

只收录能找到可靠来源、且对时间线有解释价值的事件。发布日期在当前日期之后、来源无法访问、或只有传闻的条目不作为已验证事件展示；必要时标为待确认并退出主时间轴。

## 来源与事实校验

- 优先级：官方发布或论文原文，其次为监管文件、公司工程博客与高质量一手报道。
- 每个新事件至少一个可访问来源；重大争议或关键数字尽量使用两个互相独立的来源。
- `verified` 只表示页面中的核心事实已由来源支持，不表示对事件价值判断存在唯一结论。
- 构建时继续检查内部引用；不存在的未来事件不再作为普通相关链接输出。

## 实施切片

1. 收紧 CSS 字号、行宽和垂直间距。
2. 清理页面壳层、导航、筛选、页脚与元信息文案。
3. 深改现有 30 个事件及五条模型谱系。
4. 补充高价值事件，优先填补 2012–2026 的模型与产品密集期，再补早期基础节点。
5. 同步英文内容与搜索索引。
6. 运行构建和链接检查，并在桌面、移动端、明暗主题、中英文状态下完成页面 QA。
7. 提交、推送、部署独立 Pages 项目；Worker 路由无变化时不重新部署 Worker。

## 验收标准

- 首页第一屏能看到时间轴事件，不再被超大标题占满。
- 所有页面不存在 100px 以上内容标题，移动端标题不压迫正文。
- 时间轴事件数量达到 60–80 个，且新增内容有来源。
- 重点事件详情包含具体事实或数字，不依靠宣传性结论支撑篇幅。
- 中英文切换、明暗主题、搜索、筛选和内部链接正常。
- 桌面与 390px 移动端无横向溢出、无控制台错误。
- `pnpm build` 通过，正式域名部署后核心路径返回正常。
