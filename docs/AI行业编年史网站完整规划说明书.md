# AI 行业编年史网站完整规划说明书

## 1. 产品一句话定义

**AI 行业编年史**是一个以"关键事件"为主线，以"概念、公司、模型、产品、论文、人物、技术路线"为连接的 AI 行业知识地图。

它不是单纯的新闻站，也不是工具导航，更不是冷冰冰的百科。它应该像一本可以无限翻阅、持续更新、不断生长的 AI 发展史网页书。

它要帮助读者回答这几个问题：

**AI 是怎么一步步发展到今天的？
每个标志性事件为什么重要？
这些事件背后有哪些概念值得学习？
不同公司、模型、产品、技术路线之间有什么关系？
一个刚入行的人应该怎样顺着历史脉络理解 AI 行业？**

你真正要做的不是"AI 事件列表"，而是：

> **用时间轴组织历史，用事件解释变化，用概念帮助入门，用关系图谱展示行业演化。**

---

## 2. 产品核心气质

这个网站应该有三种气质。

第一，它要有**历史感**。
用户进入网站后，不应该感觉是在看资讯流，而应该感觉自己在翻一本 AI 时代的编年史。从 AlexNet、AlphaGo、Transformer、GPT-3、ChatGPT、Copilot、Claude Code、Codex Agent，到后续各种 Agent、AI Coding、多模态、开源模型、AI Infra 的演化，都应该形成一条清晰的脉络。

第二，它要有**学习感**。
刚入行的人看到一个事件，不只是知道"发生了什么"，还要知道"这里面涉及什么概念"。比如看到 ChatGPT，就能理解 LLM、RLHF、Prompt、Token、上下文窗口、幻觉；看到 Claude Code，就能理解 Coding Agent、工具调用、代码库上下文、SWE-bench、MCP、终端代理。

第三，它要有**地图感**。
每个事件不是孤立的。它应该能连到相关公司、模型、论文、概念、产品、行业影响和后续事件。用户不是只读一篇文章，而是在一张 AI 行业知识地图里不断探索。

---

## 3. 产品不应该是什么

这个边界非常重要。

它**不是 AI 新闻站**。
新闻站追求快，而这个网站追求"有历史价值"。不是每个模型更新、每个融资、每个工具发布都值得进入主时间轴。只有那些真正改变认知、技术路线、产品范式、开发方式、行业结构的事件，才值得记录。

它**不是 AI 工具导航站**。
工具可以出现，但只有当一个工具代表了某种范式变化时才进入。例如 GitHub Copilot、Cursor、Claude Code、Codex 这类工具值得写，是因为它们代表 AI Coding 的演进，不是因为它们只是"好用工具"。

它**不是纯百科**。
百科通常只解释"是什么"。你的网站要解释"为什么重要""改变了什么""对后来有什么影响""新人应该学什么"。

它**不是纯教程网站**。
教程是辅助。你的主线不是"教你怎么用某个工具"，而是"通过关键事件理解 AI 行业"。

它**不是模型排行榜**。
排行榜变化太快，而且容易把读者带向短期比较。你的核心应该是长期脉络，而不是今天谁分数高、谁参数多。

# 4. 核心用户是谁

## 4.1 刚入行的 AI 新人

这是最重要的一类用户。

他们的问题通常是：

"为什么大家都说 Transformer 很重要？"
"ChatGPT 到底改变了什么？"
"GPT、Claude、Gemini、Llama、DeepSeek、Qwen 分别是什么关系？"
"Agent、RAG、RLHF、Token、上下文窗口这些词到底怎么理解？"
"AI Coding 是突然冒出来的吗？它之前经历了什么？"

对他们来说，网站应该是一个**可自学的 AI 行业入门地图**。

---

## 4.2 AI 产品经理、开发者、独立开发者

他们关心的是：

"现在这些 AI 产品形态是怎么演化出来的？"
"AI Coding 为什么从补全代码走向 Agent？"
"OpenAI、Anthropic、Google、Meta、DeepSeek、阿里、字节这些玩家分别在走什么路线？"
"下一个产品机会可能从哪里出现？"

对他们来说，网站应该是一个**行业趋势理解工具**。

---

## 4.3 内容创作者、研究者、投资人、从业者

他们需要：

"某个事件的原始出处是什么？"
"某条技术路线从哪里开始？"
"一个概念第一次变得重要是在哪个时间点？"
"某家公司过去几年发布过哪些关键产品和模型？"

对他们来说，网站应该是一个**可引用、可查证、可追溯的资料库**。

---

# 5. 整体信息架构

这个网站的核心不是"页面"，而是"实体关系"。

最核心的实体是：

**事件 Event**

然后每个事件连接到：

**概念 Concept**
**公司 Company**
**模型 Model**
**产品 Product**
**论文 Paper**
**人物 Person**
**主题 Topic**
**基准 Benchmark**
**数据集 Dataset**
**技术路线 Technology Path**
**来源 Source**

网站的信息结构可以理解成这样：

```text
事件
 ├── 发生时间
 ├── 相关公司
 ├── 相关模型
 ├── 相关产品
 ├── 相关论文
 ├── 相关人物
 ├── 涉及概念
 ├── 所属主题
 ├── 上游背景事件
 ├── 下游影响事件
 ├── 行业影响
 ├── 原始来源
 └── 更新记录
```

也就是说，**事件是入口，概念是学习单元，主题是理解路径，公司是行业玩家，模型和产品是技术落地，论文是源头，来源是可信度基础。**

# 6. 网站主导航设计

最终完整状态下，顶部导航可以是：

```text

主题
概念
公司
模型 / 产品
论文
学习路径
来源库
更新记录

```

每个导航的职责不一样。

**首页**负责让新用户快速理解这个网站是什么。
**时间轴**负责提供最核心的翻阅体验。
**主题**负责按方向学习，比如 AI Coding、Agent、多模态、开源模型。
**概念**负责给新人补基础知识。
**公司**负责理解行业玩家。
**模型 / 产品**负责理解具体发布物。
**论文**负责追溯技术源头。
**学习路径**负责把事件和概念组织成适合新人学习的路线。
**来源库**负责可信度和引用。
**更新记录**负责展示网站持续维护的痕迹。
**关于**负责说明你的编辑原则、收录标准和方法论。

# 9. 事件详情页设计

事件详情页是这个网站最重要的内容单元。

每个事件页面都应该回答：

发生了什么？
当时背景是什么？
为什么重要？
涉及哪些概念？
它改变了什么？
它影响了哪些后续事件？
有哪些原始来源？
新人读完应该理解什么？

---

## 9.1 事件页顶部信息

每个事件页顶部应该包含：

标题
副标题
发生日期
相关公司
相关人物
相关模型 / 产品
所属主题
重要程度
可信度状态
最后更新时间

例如：

```text
2022｜ChatGPT 发布

副标题：
AI 第一次真正进入普通人的日常生活。

日期：
2022-11-30

相关公司：
OpenAI

所属主题：
大语言模型、消费级 AI、AI 应用

重要程度：
S 级

涉及概念：
LLM、RLHF、Prompt、Token、上下文窗口、幻觉

最后更新：
2026-07-08
```

---

## 9.2 事件页正文结构

每个事件页最好固定成这个结构。

### 1. 30 秒看懂

用非常短的语言解释这个事件。

例如：

> ChatGPT 的意义不只是"AI 会聊天了"，而是它第一次让普通人用自然语言直接操作大语言模型。它把大模型从论文、API 和开发者圈，带到了大众日常生活里。

---

### 2. 发生了什么

讲事实。

谁发布的？
什么时候发布的？
发布了什么？
核心能力是什么？
当时的产品形态是什么？

这一段尽量客观，不要加入太多判断。

---

### 3. 当时的行业背景

这是很多人会忽略但非常重要的部分。

比如写 ChatGPT，就要解释：

在 ChatGPT 之前，大模型已经存在。
GPT-3 已经展示了 few-shot learning。
但普通人并没有一个足够简单、稳定、好用的入口。
聊天界面降低了使用门槛。

写 Claude Code，就要解释：

在 Claude Code 之前，AI Coding 已经经历过代码补全、IDE 助手、聊天式编程、代码库问答。
但 Claude Code 代表的是更靠近真实软件工程流程的终端代理形态。

---

### 4. 为什么重要

这是事件页的灵魂。

每个事件都要从几个维度解释：

技术上改变了什么？
产品上改变了什么？
用户行为上改变了什么？
行业竞争上改变了什么？
开发者生态上改变了什么？
它让哪些新机会出现了？

这部分可以有明确判断，但每个判断都要有理由。

---

### 5. 涉及概念

每个事件都应该有一个"概念盒子"。

例如 ChatGPT 页面：

```text
你需要理解这些概念：

LLM：大语言模型，能够基于大规模文本训练生成和理解语言。
RLHF：通过人类反馈强化学习，让模型回答更符合人类偏好。
Prompt：用户给模型的自然语言指令。
Token：模型处理文本的基本单位。
上下文窗口：模型一次能看到和处理的信息范围。
幻觉：模型生成看似合理但不准确内容的现象。
```

每个概念都可以点击进入概念页。

这个模块是你网站区分普通时间线的关键。

---

### 6. 它改变了什么

建议拆成四类影响：

**对用户的影响**
普通人开始直接使用 AI。

**对开发者的影响**
开发者开始围绕大模型 API 构建应用。

**对公司的影响**
科技公司开始重新调整 AI 战略。

**对行业的影响**
AI 从研究和后台能力变成前台产品入口。

不同事件的影响维度可以不同，但结构最好稳定。

---

### 7. 之前与之后

这是一个非常适合新人的模块。

例如：

```text
在它之前：
AI 更多存在于论文、Demo、API、企业后台能力里。

在它之后：
AI 变成普通人可以直接对话、使用、分享的消费级产品。
```

这个模块可以帮助用户快速建立历史对比。

---

### 8. 后续影响链

每个事件都要连接后续事件。

例如 ChatGPT 后续可以连接：

GPT-4
插件
GPTs
Assistants API
多模态模型
AI Agent
企业知识库
AI 搜索
AI Coding 工具

这会让用户从一个事件继续探索。

---

### 9. 争议与局限

这部分非常重要，因为它能让网站显得更可信。

比如：

ChatGPT 页面可以写幻觉、可靠性、版权、数据隐私、教育作弊等问题。
开源模型页面可以写开放权重与真正开源的区别。
Agent 页面可以写任务失败率、成本、权限、安全问题。
AI Coding 页面可以写代码质量、上下文限制、测试依赖、责任归属。

一个好的编年史不能只写高光，也要写问题。

---

### 10. 原始来源

每个事件页都应该有来源区。

来源可以分层：

官方发布
论文
技术报告
产品文档
GitHub 仓库
发布会视频
权威媒体报道
行业分析
社区讨论

最好明确标注：

**主要来源**
**补充来源**
**延伸阅读**

这样用户会信任你。

---

### 11. 更新记录

每个事件页底部应该有更新记录。

例如：

```text
2026-07-08：补充 Claude Code 与 Codex Agent 的相关链接。
2026-06-20：修订 RLHF 概念解释。
2026-05-12：新增原始发布来源。
```

这会让网站有"活档案"的感觉。

## 10.2 概念页模板

以 RLHF 为例：

```text
RLHF
Reinforcement Learning from Human Feedback

一句话解释：
RLHF 是一种让模型通过人类反馈变得更符合人类偏好的训练方法。

新人解释：
大模型刚训练出来时，可能会生成很多看似合理但不符合人类期待的回答。RLHF 的作用，是让人类对不同回答进行偏好标注，再让模型学习"什么样的回答更好"。

为什么重要：
它让聊天式大模型更容易被普通用户使用，也是 ChatGPT 这类产品能够形成良好对话体验的重要原因之一。

相关事件：
InstructGPT
ChatGPT
GPT-4
Claude

相关概念：
Instruction Tuning
Reward Model
Alignment
Constitutional AI
```

# 18. 事件收录标准

这是整个项目的编辑灵魂。

你必须有一套明确标准，否则网站会变成新闻堆。

## 18.1 一个事件是否值得收录，看六个维度

### 1. 范式变化

它是否改变了大家理解 AI 的方式？

例如 ChatGPT 改变了大众对 AI 的认知。
GitHub Copilot 改变了开发者对代码生成的认知。
Claude Code 代表 AI Coding 从辅助走向更完整任务执行。

### 2. 技术影响

它是否带来了新的技术路线、架构、训练方法、工程方法？

例如 Transformer、RLHF、Diffusion、MoE、RAG、Tool Calling、MCP。

### 3. 产品影响

它是否创造了新的产品入口？

例如 ChatGPT 的聊天入口，Copilot 的 IDE 入口，Claude Code 的终端入口。

### 4. 生态影响

它是否带动了开发者、创业公司、开源社区、平台生态？

例如 Hugging Face、Llama、Stable Diffusion、vLLM、Ollama、Cursor。

### 5. 持续影响

它是否在发布后持续影响后续几年？

短期热点不一定值得进主时间轴。
真正的标志性事件应该有后续影响链。

### 6. 概念密度

它是否适合帮助新人理解一个重要概念？

有些事件本身可能不够大，但它非常适合解释一个关键概念，也值得收录。

---

## 18.2 重要程度分级

建议使用 S / A / B / C 四级。

### S 级：范式级事件

改变整个行业方向，长期影响巨大。

例如：

Transformer
ChatGPT
GPT-4
Stable Diffusion
Llama 系列代表的开放模型路线
AI Coding Agent 的关键转折事件

### A 级：行业级事件

对某个重要领域产生明显影响。

例如：

GitHub Copilot
Claude Code
Devin
Assistants API
Function Calling
重要开源模型发布
重要多模态模型发布

### B 级：领域级事件

对某个细分方向重要。

例如某个 benchmark、某个工具、某个框架、某个产品的重要版本。

### C 级：补充事件

适合做背景资料，但不一定放在主时间轴最显眼位置。

---

## 18.3 事件状态

每个事件可以有状态：

**已确认**
有可靠来源，事实明确。

**待补充**
事件存在，但资料不完整。

**有争议**
行业对其意义或事实存在争议。

**已过时**
曾经重要，但后续影响下降。

**背景事件**
不是核心节点，但有助于理解主线。

# 25. 可信度与编辑原则

这个网站要长期有价值，最重要的是可信度。

## 25.1 事实和判断分离

每篇内容里要区分：

**事实**
某年某月某日，某公司发布了某产品。

**解释**
这个产品降低了普通用户使用大模型的门槛。

**判断**
这个事件是 AI 消费级爆发的重要节点。

**预测**
这个方向可能继续影响未来软件开发流程。

事实要有来源。
解释要有逻辑。
判断要有理由。
预测要明确标注为预测。

---

## 25.2 不追短期热度

不要因为某个事件在社交媒体上很火，就立刻把它写成历史节点。
可以先放入候选池观察。

这个网站的价值来自"筛选"，不是"搬运"。

---

## 25.3 保留争议

有争议的事件不要回避。

比如：

开放权重是否等于开源？
AI Agent 到底有多少真实可用性？
AI Coding 是提升效率还是制造技术债？
模型评测是否能代表真实能力？
多模态能力是否被过度营销？
AI 安全讨论是否被商业竞争影响？

这些争议会让网站更像真实的行业史，而不是宣传稿。

用户是来翻历史的，但真正留下来，是因为他学到了东西。

---

## 4. 事件页要写多深？

建议分层。

顶部给 30 秒摘要。
中间给完整解释。
底部给来源和延伸阅读。
新人可以浅读，专业读者可以深挖。

---

## 5. 概念解释要多技术？

建议采用双层解释。

先用人话解释。
再给技术解释。
再给例子。
最后给相关论文和事件。

这样不会吓跑新人，也不会显得浅。

---

## 6. 如何避免变成流水账？

每个事件必须回答"为什么重要"。

没有"为什么重要"的事件，不进入主时间轴。

---

## 7. 如何避免变成主观评论？

每个判断后面要有依据。

比如不要只写：

"这是一个划时代产品。"

要写：

"它的重要性在于，它第一次让普通用户通过聊天界面大规模使用大语言模型，并推动了后续 AI 应用生态的爆发。"

---

## 8. 如何处理快速变化的信息？

给每个页面加：

最后更新时间
内容状态
来源更新时间
更新记录
争议说明

这样即使行业变化快，你的网站也不会显得过时。

---

## 9. 如何形成壁垒？

壁垒不是"你也写了 ChatGPT"。

壁垒在于：

事件之间的关系
概念解释的质量
收录标准的稳定
来源体系的可信
主题路线的组织能力
长期更新的耐心

---

## 10. 如何让用户愿意反复回来？

靠几个东西：

最近更新
历史上的今天
每周新增事件
专题持续完善
Newsletter
可收藏事件
学习路径进度
站内 AI 问答
年度回顾

---

## 11. 如何让内容可复用？

每个事件都应该能被复用成：

网页文章
短视频脚本
推文
公众号文章
知识卡片
课程材料
Newsletter
时间轴截图

这样你写一份内容，可以服务多个渠道。

---

## 12. 这个项目最终的品牌是什么？

我建议品牌不要太窄。

不要只叫"大模型编年史"，因为未来 AI 会包含 Agent、机器人、多模态、AI 科学、AI 硬件、AI Infra。

更推荐：

**AI Chronicle｜AI 行业编年史**

中文副标题：

**用关键事件理解 AI 的发展脉络。**

或者：

**给刚入行的人看的 AI 发展地图。**

# 36. 我最建议你坚持的核心原则

最后给你一个很重要的判断：

这个项目最有价值的地方，不是"记录得全"，而是"解释得清楚"。

你不需要收录每一条 AI 新闻。
你需要筛出那些真正值得被历史记住的节点。

每个节点都要做到：

**让新人看懂。
让从业者觉得有脉络。
让创作者可以引用。
让未来回头看时仍然有价值。**

这个网站如果做好，它会成为你的长期内容资产，也会成为一个非常适合新人入门 AI 行业的公共知识库。
