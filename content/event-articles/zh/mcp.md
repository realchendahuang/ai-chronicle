一个 AI 应用接入一个数据源，通常不算难事。十个应用各自接入十个数据源，事情便不再只是二十个端点和几段认证代码。每个宿主有自己的插件格式，每项服务有自己的对象和权限，每一对连接都要重新解释：这里有什么可读，那边允许做什么，结果以什么形式返回。

2024 年 11 月 25 日，Anthropic 发布 Model Context Protocol（MCP），针对的是这道乘法题。

Client 与 Server 的分工使生态可以沿两条线生长：宿主实现 Client/Host，工具提供者实现 Server。示例服务器覆盖文档、聊天、代码与浏览器自动化，展示的是“协议能表达的工作类型”，不是“这些集成已经企业级完备”。早期采用者名单是启动社会资本，真正的标准压力来自第二个、第三个宿主是否用同一握手而不加私有方言。安全上，本地 Server 的文件系统与命令权限尤其危险，默认信任任何社区 Server 会重演插件时代的教训。MCP 的赌注是：连接税可以下降，只要信任税被认真收取。

协议采用客户端—服务器结构：MCP Server 暴露能力，Client 与它建立连接、协商协议，Host 则承载模型、用户交互与权限决定。Server 可以提供工具、只读资源和提示模板；消息层使用 JSON-RPC。这样，一个 GitHub、Postgres 或本地文件集成有机会被多个兼容宿主复用，而不必为每一种模型界面重写。Anthropic 同时开放规范、SDK、Claude Desktop 的本地服务器支持，以及一组示例服务器。

这与函数调用解决的不是同一个问题。函数调用约定模型怎样选择工具、怎样生成参数；MCP 更关心工具从哪里被发现，宿主怎样连接它，能力如何描述，结果如何在一套通用接口中往返。前者像一句调用语法，后者试图规定插座。两者可以一起工作，也都不决定插座后面接着什么。把 MCP 写成“又一个 tool use 按钮”，会漏掉它真正想标准化的层。

发布之初，Anthropic 列出了 Google Drive、Slack、GitHub、Git、Postgres 和 Puppeteer 等预制服务器，并提到 Block、Apollo 以及 Zed、Replit、Codeium、Sourcegraph 等早期采用者或合作方。这些是当时的发布参与和计划，不等于协议已经成为行业标准。标准的生命要由后来有多少独立实现、不同宿主是否保持兼容、规范能否在扩展中不碎裂来决定，发布者不能单方面宣布。

而连接从来不只是便利。一个 Server 如果能读文件、查数据库、执行命令，它也可能返回恶意内容，诱导模型调用别的工具，或获得用户并未真正理解的权限。MCP 规定通信边界，不替宿主完成信任判断。授权提示、最小权限、来源标识、结果隔离和对提示注入的防护，仍落在实现者与用户身上。通用插座让设备更容易接入，也让接错设备的后果更值得认真对待。

协议史很少像模型发布那样拥有耀眼的演示。它的成功往往表现为某段胶水代码不再需要写，某个集成换了宿主仍能工作，甚至最后没人再注意协议本身。MCP 所争取的正是这种隐身：让 AI 与外部世界的连接不必每一次都从私人约定开始。2024 年底交出的是规范与首批实现，不是终局生态——JSON-RPC 消息与 tools/resources/prompts 三类能力，使 MCP 看起来更像“给 LLM 宿主用的 LSP”，而不是又一个应用内插件 API。Claude Desktop 本地服务器降低了试玩成本；开源规范则邀请其他宿主实现同一套握手。安全问题随之从“模型会不会胡写参数”扩展到“用户是否安装了不该信任的 Server”。协议层成功时，人们会忘记协议；失败时，人们会发现权限对话框与默认信任策略比 schema 字段更致命。

与函数调用相比，MCP 把“工具安装与发现”拉到模型对话之外的系统层：用户或管理员先决定信任哪些 Server，模型再在已暴露的工具中选择。这种分层接近操作系统权限模型，也因此继承了供应链风险——恶意 Server 可以是新的攻击面。协议史写到这里，必须把便利与攻击面写在同一句。

发布当日的材料是规范、SDK 与示例，而不是强制全行业切换的法令。是否成为默认连接层，要看后续独立实现是否愿意保持兼容，以及权限 UX 是否让普通人看得懂自己在授权什么。

2024 年底交出的是规范与首批实现，不是终局生态——后者要用兼容性与时间来换。
