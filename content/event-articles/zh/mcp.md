一个 AI 应用接入一个数据源，通常不算难事。十个应用各自接入十个数据源，事情便不再只是二十个端点和几段认证代码。每个宿主有自己的插件格式，每项服务有自己的对象和权限，每一对连接都要重新解释：这里有什么可读，那边允许做什么，结果以什么形式返回。

2024 年 11 月 25 日，Anthropic 发布 Model Context Protocol（MCP），针对的是这道乘法题。

协议采用客户端—服务器结构：MCP Server 暴露能力，Client 与它建立连接、协商协议，Host 则承载模型、用户交互与权限决定。Server 可以提供工具、只读资源和提示模板；消息层使用 JSON-RPC。这样，一个 GitHub、Postgres 或本地文件集成有机会被多个兼容宿主复用，而不必为每一种模型界面重写。Anthropic 同时开放规范、SDK、Claude Desktop 的本地服务器支持，以及一组示例服务器——覆盖文档、聊天、代码与浏览器自动化等，展示的是「协议能表达的工作类型」，不是「这些集成已经企业级完备」。

这与函数调用解决的不是同一个问题。函数调用约定模型怎样选择工具、怎样生成参数；MCP 更关心工具从哪里被发现，宿主怎样连接它，能力如何描述，结果如何在一套通用接口中往返。前者像一句调用语法，后者试图规定插座。两者可以一起工作，也都不决定插座后面接着什么。把 MCP 写成「又一个 tool use 按钮」，会漏掉它真正想标准化的层。

发布之初，Anthropic 列出了 Google Drive、Slack、GitHub、Git、Postgres 和 Puppeteer 等预制服务器，并提到 Block、Apollo 以及 Zed、Replit、Codeium、Sourcegraph 等早期采用者或合作方。这些是当时的发布参与和计划，不等于协议已经成为行业标准。标准的生命要由后来有多少独立实现、不同宿主是否保持兼容、规范能否在扩展中不碎裂来决定，发布者不能单方面宣布。

连接从来不只是便利。一个 Server 如果能读文件、查数据库、执行命令，它也可能返回恶意内容，诱导模型调用别的工具，或获得用户并未真正理解的权限。MCP 规定通信边界，不替宿主完成信任判断。授权提示、最小权限、来源标识、结果隔离和对提示注入的防护，仍落在实现者与用户身上。与函数调用相比，MCP 把「工具安装与发现」拉到模型对话之外的系统层：用户或管理员先决定信任哪些 Server，模型再在已暴露的工具中选择。这种分层接近操作系统权限模型，也因此继承了供应链风险——恶意 Server 可以是新的攻击面。

协议史很少像模型发布那样拥有耀眼的演示。它的成功往往表现为某段胶水代码不再需要写，某个集成换了宿主仍能工作，甚至最后没人再注意协议本身。2024 年底交出的是规范与首批实现，不是终局生态——后者要用兼容性与时间来换。便利与攻击面，必须写在同一句。
