工具调用开始成为行业共同能力时，更换模型已经不会让开发者面对一套完全陌生的工作。

先写下工具名称和说明，再用 JSON Schema 约束参数；模型根据用户请求选择一个或多个工具，把结构化参数交还应用；应用执行数据库查询、业务 API 或其他操作，把结果送回模型。2024 年 5 月 30 日，Anthropic 宣布 Claude Tool Use 正式可用（GA）时，这套循环已经有了熟悉感。它从更早的 beta/预览阶段走到生产默认支持，细节仍带着 Claude 的内容块约定——例如以 `tool_use` 等形式返回结构化调用——但熟悉并不意味着与 OpenAI 函数调用在提示、错误处理、并行调用和返回格式上完全一致。语言模型与传统软件的分工正在稳定下来，平台方言还在。

在 Tool Use GA 之前，开发者若要在 Claude 上实现类似能力，往往依赖更脆弱的约定输出或半正式 beta。GA 把 `tool_use` 内容块、错误与多轮回传写成稳定合同，使评估基准与生产日志可以按同一字段解析。视觉工具选择扩展了“工具”的定义：工具不必只服务纯文本客服，也可以服务截图、单据与界面。Anthropic 没有在这一天宣布跨宿主标准，它宣布的是：Claude 作为模型端，愿意稳定地说出“我要调用什么”。插座标准是下一章的事。

Claude 可以在一次请求里选择多个工具，也能在包含图像的任务中判断需要调用什么。Anthropic 给出的客服和图像示例，让工具使用离开抽象的 Agent 演示，落到查询订单、读取信息、结合视觉材料继续处理这些普通工作上。模型负责理解不整齐的人类请求，工具负责接触实时数据和受控操作。两者之间仍由应用把关：模型输出调用请求，并不直接获得数据库或账户的权限。GA 的意义在于接口与行为被当作稳定产品表面交付，而不是“演示里偶尔能用”。

当这种能力只存在于一家 API 时，开发者会把它视作产品特性；当多家前沿模型都提供相似入口，它便开始像一层基础设施。Agent 框架可以尝试在模型之间迁移，工具定义也不必每次从纯文本暗号重新开始。与此同时，差异会变得更刺眼：同一个工具要怎样描述，结果怎样返回，鉴权与上下文由谁管理，服务又怎样向陌生客户端声明自己能做什么？

Tool Use 没有回答后面所有问题。它只定义 Claude 与客户端工具之间的一次调用方式，并未形成跨模型、跨服务的统一协议。但重复劳动已经清楚可见：越来越多平台都要描述工具、选择工具、传递参数、带回结果，同一个集成却仍需按供应商重写。数月后 Anthropic 发布的 MCP，会尝试处理宿主与外部系统之间的发现与连接问题；Tool Use 让“模型如何选工具”变得具体，却不是那套标准的预告片，两者解决的层次不同，不应混写。

这次正式可用不靠某个惊人的演示成立。它留下的是一个安静的工程事实：概率模型开始在不同厂商那里遵守相似的接口边界。开发者换掉模型以后，工具定义、结构化参数与结果回传仍然是熟悉的工作；至于连接层何时能够复用，GA 宣布还意味着文档、错误码与最佳实践开始按生产标准维护，而不是停留在“实验性能用”。多工具并行与视觉场景下的工具选择，把 Claude 推进到与同业可比的 Agent 基础设施层；开发者终于可以在评估中把“会不会用工具”当成默认能力，而不是某家独家彩蛋。与此同时，客户端仍必须实现执行器：模型返回 `tool_use` 块之后，真正的 SQL、HTTP 与副作用仍在应用进程里。边界清楚，是这套语法能跨团队复制的原因。

当 OpenAI 函数调用与 Claude Tool Use 并存，框架作者开始写适配层：同一业务工具，两套请求字段。这种适配层的出现，本身就是“共同语法尚未统一协议”的证据。Tool Use GA 让适配成为值得做的事，因为 Claude 一侧终于足够稳，值得为之维护。

开发者换掉模型以后，工具定义、结构化参数与结果回传仍然是熟悉的工作；至于连接层何时能够复用，还要等下一套协议来回答。
