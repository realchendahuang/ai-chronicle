工具调用开始成为行业共同能力时，更换模型已经不会让开发者面对一套完全陌生的工作。

先写下工具名称和说明，再用 JSON Schema 约束参数；模型根据用户请求选择一个或多个工具，把结构化参数交还应用；应用执行数据库查询、业务 API 或其他操作，把结果送回模型。2024 年 5 月，Anthropic 宣布 Claude Tool Use 正式可用时，这套循环已经有了熟悉感。熟悉并不意味着细节完全一致。提示写法、错误处理、并行调用和返回格式仍带着不同平台的习惯，但语言模型与传统软件的分工正在稳定下来。

Claude 可以在一次请求里选择多个工具，也能在包含图像的任务中判断需要调用什么。Anthropic 给出的客服和图像示例，让工具使用离开抽象的 Agent 演示，落到查询订单、读取信息、结合视觉材料继续处理这些普通工作上。模型负责理解不整齐的人类请求，工具负责接触实时数据和受控操作。两者之间仍由应用把关：模型输出调用请求，并不直接获得数据库或账户的权限。

当这种能力只存在于一家 API 时，开发者会把它视作产品特性；当多家前沿模型都提供相似入口，它便开始像一层基础设施。Agent 框架可以尝试在模型之间迁移，工具定义也不必每次从纯文本暗号重新开始。与此同时，差异会变得更刺眼：同一个工具要怎样描述，结果怎样返回，鉴权与上下文由谁管理，服务又怎样向陌生客户端声明自己能做什么？

Tool Use 没有回答后面所有问题。它只定义 Claude 与客户端工具之间的一次调用方式，并未形成跨模型、跨服务的统一协议。但重复劳动已经清楚可见：越来越多平台都要描述工具、选择工具、传递参数、带回结果，同一个集成却仍需按供应商重写。后来包括 MCP 在内的协议工作，会尝试处理这类连接问题；Tool Use 让问题变得具体，却不是那套标准的预告片。

这次正式可用不靠某个惊人的演示成立。它留下的是一个安静的工程事实：概率模型开始在不同厂商那里遵守相似的接口边界。开发者换掉模型以后，工具定义、结构化参数与结果回传仍然是熟悉的工作；至于连接层何时能够复用，还要等下一套协议来回答。
