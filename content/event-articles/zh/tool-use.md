2024 年 5 月 30 日，Anthropic 宣布：Claude 的 Tool Use 正式可用。

这句话听起来像版本号升级，实际钉住的是一条分工。模型可以提出“我要调用哪个工具、参数是什么”；真正去查库、改订单、发请求的，仍是应用。参数用 JSON Schema 约束，调用以内容块（例如 `tool_use`）返回——从 beta 走到生产默认支持，语法更稳了，责任边界没有挪走。把模型返回的参数直接当已授权操作，是集成错误，不是模型“学会了执行”。

它与 OpenAI 一年前的函数调用属于同一家族，却不是同一方言：提示写法、错误处理、并行与返回格式并不完全互通。客服查单、读图后再调工具等样例写在发布材料里，意图很清楚——模型负责理解与选型，客户端负责权限、超时、副作用与审计。

正式可用意味着文档、稳定性与示例面向生产集成，而不是演示脚本。工具循环一旦成为默认，失败模式也默认化：错误 schema、幻觉参数、该问人却先调工具。产品要把拒绝、澄清与重试写进同一条路径。Tool Use 也不是后来的 Model Context Protocol。前者规定一次对话里如何发出结构化调用；后者试图规定宿主、客户端与服务器如何发现和连接工具。时间线与抽象层不同。

2024 年春天的 GA，只是把“模型输出可执行意图”这一环钉进 Claude 的默认产品面。协议前史往往如此：共同语法先出现，统一标准后到。方言仍在的时候，工程师的工作是翻译与防御；标准到来时，工作变成兼容与治理——问题并没有消失，只是换了一层名字。

当 Tool Use 成为默认，集成测试必须覆盖“模型选错工具”“参数类型不合”“工具超时”等路径。协议前史的意义，是让这些路径有名字、有内容块、有文档。MCP 等后续协议试图解决发现与连接；Tool Use 解决的是会话内如何说话。两层齐备之前，工程师一直在翻译方言。
