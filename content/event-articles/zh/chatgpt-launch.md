ChatGPT 的发布文章没有为它安排盛大的舞台。2022 年 11 月 30 日，OpenAI 的开头只说：我们训练了一个可以用对话方式交互的模型，叫 ChatGPT。研究预览免费，旁边是一条试用链接。没有现场演示的倒计时，没有参数量口号，也没有要求读者先读完一篇论文。

界面做的减法近乎彻底。用户输入自然语言，模型回答；回答不对，可以接着指出问题，不必把全部背景重新写一遍。对话历史让“解释得简单一点”“换一种写法”“你刚才漏了什么”成为完整指令。此前，同类能力多半藏在 API 调用、研究演示或需要自己维护状态的脚本里；这一次，GPT-3.5 与 InstructGPT 训练路线上的成果，被装进一种人人熟悉的形式：消息往来。

开放当天没有附带权重下载，也没有要求信用卡。任何人只要通过账号与地区、政策允许的路径进入，就能在浏览器里开始对话；服务端的速率与容量限制会决定排队与拒绝，而不是一张功能完整的 SLA。知识截止日期、无法联网（首日）、以及安全策略造成的误拒与漏拒，都是同一产品页上与能力宣传并列的文字。后来的增长故事常被拿来定义这一天，可增长曲线并不是发布页上的内容；发布页上的内容是界面、示例错误、局限清单与反馈入口。把事件写准确，就要把这些“不好看”的部分留在正文里。

发布页上的代码示例恰好保留了这种产品的两面。用户贴出一段 Go 代码，说错误没有传出来，怀疑 channel 有问题。ChatGPT 先索取更多上下文；用户补充后，它判断 `resultWorkerErr` 没有关闭，建议写入后关闭 channel。然而原代码已经有 `defer close(resultWorkerErr)`。模型的回应语气流畅、步骤完整，关键观察却错了。错误没有藏在压力测试附录里，而是与产品一起出现在介绍页面上。

OpenAI 对局限的罗列同样直接：模型会生成听来可信但错误或荒谬的答案；轻微改写问题可能得到不同结果；它过度冗长，经常重复自己是 OpenAI 训练的语言模型；面对含混要求，常常猜测而不是追问；安全拒绝既有漏网，也有误伤。页面解释了一项难处：强化学习阶段没有自动提供事实真值，鼓励谨慎又可能让模型拒绝原本会答的问题。知识也有边界——训练数据截止后的新闻、实时状态、个人账户里的信息，它都无法可靠掌握。

“聊天”早已进入训练过程。训练者曾同时扮演用户与助手，写出对话；系统再为同一轮对话采样多个回答，由人排序，训练奖励模型，经过数轮 PPO 调整。页面保留点赞、点踩和问题反馈入口，甚至用 API 额度奖励反馈竞赛。产品向公众开放能力的同时，也把公众交互接回了改进模型的数据回路。ChatGPT 不是把一个封闭的“完成品”抛向市场，而是把研究预览本身当成持续收集信号的入口。

此前，调用相近能力往往意味着申请 API、写代码、处理请求与响应、自己决定如何保存会话。ChatGPT 把这些环节折叠成打开网页、打一行字。教育者、程序员、写作者和普通好奇者无需先决定自己属于哪一种“AI 用户”，可以在同一会话里试问、反驳、重写和分享结果。开放范围也是有限的：免费研究预览，账号与速率受平台约束；完整 API、付费档、插件、浏览与自定义助手都不在首日清单上。用户不必先理解 token、上下文窗口或 RLHF，也能开始使用；这三个词却已经写在训练与产品设计里。反馈按钮把点赞与点踩接回改进回路，使“研究预览”名副其实：公众既是用户，也是持续标注的一端。

它的日常性正是事件的核心。没有新论文要求读者先理解，没有权重文件要求开发者先部署，也没有冠军赛规定观看时间。一个不稳定、会犯错、却足够配合的模型停在浏览器里，等人继续发下一条消息。发布页已经诚实写出了它会答错；后来发生的事情，是无数人仍愿意再问一次——不是因为错误消失了，而是因为继续追问的成本低到几乎可以忽略。
