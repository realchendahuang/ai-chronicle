GitHub Copilot 从尚未写完的代码进入编辑器。

2021 年 6 月 29 日的技术预览把 OpenAI Codex 放进 Visual Studio Code。插件读取正在编辑的代码，根据函数名、注释和周围上下文提出下一行，或者补出整个函数。开发者不需要先离开编辑器、组织一个完整问题，再把答案复制回来；建议在打字过程中出现，随后被接受、拒绝或继续修改。

这点产品设计比任何演示样例都重要。过去的自动补全擅长列出当前作用域里的变量、方法与类型，语言服务器可以精确知道某个 API 接受哪些参数。Copilot 提出的却可能是一个实现：测试用例、样板代码、常见算法，甚至一条从注释推导出的函数体。它未必知道项目的真实意图，只是把意图的迹象——名称、注释、前后代码——变成候选代码。

GitHub 在发布文中称它为“AI pair programmer”，但当时产品的实际动作更窄，也更容易进入习惯。它不参与需求讨论，不为合并后的故障负责；它只是不断给出下一种可能。预览阶段对多种语言和框架开放，尤其强调 Python、JavaScript、TypeScript、Ruby 与 Go。名额有限，能力仍带着试验性质。

“建议”这个身份既降低了使用门槛，也安排了责任。代码没有因为来自模型就自动正确，开发者仍需检查依赖、边界条件、安全性和许可风险。Codex 的训练数据含有大量公开源代码，这很快使训练代码的授权、相似片段输出与作者归属成为争论。编辑器里的短暂便利，连接着编辑器之外一整套尚未解决的问题。

Copilot 没有在 2021 年一次性宣布新的编程范式。它先改变了更细小的节奏：写下一行之前，屏幕上已经有一个可以反对的答案。开发者有时采纳，有时删掉，有时只取其中两行。人与模型的分工并不稳定，却通过这些频繁而不起眼的决定进入了日常工作。
