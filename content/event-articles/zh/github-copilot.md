GitHub Copilot 从尚未写完的代码进入编辑器。

2021 年 6 月 29 日的技术预览把 OpenAI Codex 放进 Visual Studio Code。插件读取正在编辑的代码，根据函数名、注释和周围上下文提出下一行，或者补出整个函数。开发者不需要先离开编辑器、组织一个完整问题，再把答案复制回来；建议在打字过程中出现，随后被接受、拒绝或继续修改。开放范围是技术预览：名额有限，先进入 VS Code，而不是所有 IDE 的默认可选功能。

把 Codex 放进 VS Code 的灰字建议，是一次交互设计上的降维：不要求用户学会提示词，只要继续写代码。接受率、拒绝率与修改率成为比基准分数更贴近产品的信号，尽管早期公开材料并没有把这些内部指标全部展开。GitHub 与 OpenAI 的合作结构也写进事件：模型能力来自一侧，分发与开发者关系来自另一侧。许可与训练数据争议随后爆发，说明“建议”再轻，也压着版权与开源社区的重问题。技术预览的轻，和法律讨论的重，从第一天就叠在同一条时间线上。

这点产品设计比任何演示样例都重要。过去的自动补全擅长列出当前作用域里的变量、方法与类型，语言服务器可以精确知道某个 API 接受哪些参数。Copilot 提出的却可能是一个实现：测试用例、样板代码、常见算法，甚至一条从注释推导出的函数体。它未必知道项目的真实意图，只是把意图的迹象——名称、注释、前后代码——变成候选代码。建议以内联灰字出现，一次 Tab 看起来很轻。

GitHub 在发布文中称它为“AI pair programmer”，但当时产品的实际动作更窄，也更容易进入习惯。它不参与需求讨论，不为合并后的故障负责；它只是不断给出下一种可能。预览阶段对多种语言和框架开放，尤其强调 Python、JavaScript、TypeScript、Ruby 与 Go。GitHub 同时明确提示：生成代码可能错误，也可能复现训练数据中的模式。那不是附带的免责声明装饰，而是产品机制与训练数据来源共同写下的警告。

“建议”这个身份既降低了使用门槛，也安排了责任。代码没有因为来自模型就自动正确，开发者仍需检查依赖、边界条件、安全性和许可风险。Codex 的训练数据含有大量公开源代码，这很快使训练代码的授权、相似片段输出与作者归属成为争论。编辑器里的短暂便利，连接着编辑器之外一整套尚未解决的问题。后来的研究与诉讼会继续拉扯这些边界；在 2021 年夏天，问题已经写在产品说明里。

2021 年的 Copilot 仍是技术预览，而且只先进入 Visual Studio Code。它的产品机制却已定型：建议以内联形式出现，开发者逐次接受、忽略或改写。一次 Tab 看起来很轻，测试、审查、授权和合并后的后果仍全部留在按键的另一边。后来的 Chat、Agent 与多文件编辑会扩展产品形态；这一天确立的是最小单位——技术预览的另一层含义是：GitHub 与 OpenAI 仍在观察真实仓库里的接受率、错误模式与安全问题。语言服务器补全与生成式补全开始共享快捷键心智，却服从不同的正确性来源——前者靠符号与类型，后者靠统计与上下文相似。开发者迅速学会“先看再 Tab”，也迅速把审查负担从“有没有建议”转成“建议是否悄悄引入漏洞或许可风险”。2021 年夏天写下的产品机制，成为此后十年 AI 编程工具的最小公约数。

预览名额与 IDE 限制，使早期采用者画像偏向 VS Code 用户与能够申请到资格的开发者。他们的使用方式——何时接受、何时改写——会通过口碑塑造“AI 编程”的第一印象。第一印象一旦形成，后来的 Chat 与 Agent 功能都要在“补全是否靠谱”的阴影下证明自己。2021 年 6 月的灰字，是整棵树的根。

从预览到后来的正式商业化，Copilot 会换模型、加聊天、进更多 IDE，但接受/拒绝/改写的基本循环保持稳定。循环稳定，意味着责任分配稳定：模型提议，人签字。2021 年技术预览把这一定律写进编辑器；此后所有更强的 Agent 叙事，仍要回到这一定律，才能解释代码为何能进主分支。

这一天确立的是最小单位——在输入流中不断给出、随时可被否决的下一块代码。
