本地终端里的编程 Agent 占据一段连续时间。人提出任务，看它搜索与修改，遇到岔路便介入。OpenAI 在 2025 年 5 月发布的云端 Codex 换了一种时间安排：任务被送走，各自进入隔离环境，几项工作可以同时进行，完成后再带着补丁和证据回来。

连接 GitHub 仓库后，Codex 会为每个任务准备预装代码的容器，在其中读取文件、运行命令、修改代码并执行测试。首版由 codex-1 驱动。一个 Agent 看不到另一个任务正在改什么，每项工作拥有自己的环境与过程；隔离既让并行成为可能，也把任务边界写得更清楚。修一个 bug、补一组测试、解释一段代码，可以成为三份相互独立的委托，而不是在一个对话里排队等待。

云端没有把软件工程变成一句命令。它只是把许多中间时刻从人的桌面移走。Agent 仍可能误读需求，测试仍可能覆盖不足，仓库里未写下的约定仍可能被忽略。OpenAI 因此把日志、测试结果与代码引用放在交付物里，让使用者能够追查它做过什么，而不是只接收一段完成宣言。任务在后台运行，责任没有一同被托管。

这与终端 Agent 形成了并非互斥的分工。终端适合边做边校正，尤其当目标尚在变化；云端任务适合边界较清楚、能够隔离、最后可以验收的工作。前者像共用一张工作台，后者更像提交一叠工单。模型能力可能相近，协作制度却不同：一个强调同步上下文，一个强调异步证据。

Codex 把队列管理带进了日常开发：哪些任务足够独立，哪些可以同时开始，什么结果算通过，冲突如何回到主分支处理。当多个 Agent 一起工作，生成代码的时间未必还是最稀缺的东西。把工作切对、把验收写清楚，以及在一批看似完成的补丁中知道哪一份值得合并，逐渐占据更重的位置。首版研究预览交付的并非“并行工程师”，而是一种更克制的产品单位：带有独立环境、过程记录与返回包的软件任务。
