GPT-3 写出一段文字，影响世界仍要经过读者。Codex 写出一段代码，代码本身可以继续调用别的东西。

2021 年 8 月 10 日，OpenAI 发布 Codex API 私有测试版时，抓住的正是这条边界。模型继承 GPT-3，又在公开来源的数十亿行代码上继续训练，包含公开 GitHub 仓库。它最擅长 Python，也覆盖 JavaScript、Go、Perl、PHP、Ruby、Swift、TypeScript 与 Shell 等十多种语言。用于 Python 时，官方称其可利用约 14KB 上下文，而 GPT-3 是 4KB。评测上，早期约 12B 参数的 Codex 在 HumanEval 单次采样中解出 28.8% 的题目——这是论文与官方材料中的具体数字，说明代码生成已可测量，也说明大多数题仍未解出。

公开代码训练带来能力，也带来许可与归属的阴影：生成结果是否会复现训练片段，是否构成风险，在 2021 年已进入讨论，并在 Copilot 产品线持续发酵。私有 beta 审核应用，是把可执行代码模型当成有滥用面的接口，而不是无害的文本补全。HumanEval 成为此后代码模型的公共尺子之一，却也固化了“函数级、单文件、有断言”的考试想象，与仓库级工程仍有距离。Codex 站在补全与 Agent 之间的过渡点：它会写可运行的东西，但默认仍要人把东西嵌进系统。

发布页面上的演示并不局限于“写一个函数”。自然语言指令被译成代码，代码再操作现成的应用与 API：做游戏、处理数据、控制软件界面。模型无需为每个应用重新学习一种专门的控制协议；只要那个应用已经向程序暴露接口，生成正确代码就可能成为通往它的中间层。这也是它与 GitHub Copilot 的关系：Copilot 把 Codex 能力放进编辑器补全；Codex API 则把同类模型交给开发者嵌入自己的产品与工作流。两条产品线共享模型族，交付形态不同。

自然语言允许含混，代码必须满足语法、类型、依赖和运行环境；一段看起来合理的程序可能调用不存在的方法，也可能在边界条件下失败。执行能力放大了用途，也放大了审查的必要。OpenAI 以私有测试和免费初期额度逐步开放 API，审核应用并观察开发者怎样使用，而没有把它当成可以放任运行的通用操作者。训练数据来自公开代码，也把许可、相似片段输出与归属问题带进讨论——这些问题在 Copilot 产品线中会变得更尖锐。

在编程内部，Codex 最先触及的是一种常见摩擦：需求已经想清楚，开发者却仍要把它拆成库、函数和调用方式。官方将编程的一部分概括为“把简单问题映射到已有代码”，并认为这是模型最擅长、也最乏味的环节。它还能做代码解释、语言转换和重构，但效果随任务而变。

这个版本的 Codex 后来在 2023 年被弃用；2025 年出现的同名命令行工具和云端软件工程 Agent 属于另一代产品，不应与 2021 年的 API 模型混为一谈。2021 年的 Codex 把代码明确放到语言模型的行动出口上：一句要求可以生成可执行的中介，再借已有 API 抵达其他软件。此后的工具调用与编程 Agent 增加了状态、权限和验证，却始终要处理这道接缝——含混的人话到了可执行的一侧，HumanEval 上的 28.8% 单次通过率常被用来同时讲述两件事：代码模型开始“能考”；以及大多数题仍失败，自动生成距离可靠工程很远。私有测试版与应用审核，说明 OpenAI 当时更担心可执行代码被滥用，而不是急于无限制开放。Copilot 让大众在编辑器里先遇到 Codex 族能力；API 则让工具制造者把同一类模型嵌进 IDE、教育产品与内部自动化。2023 年弃用旧 Codex 接口之后，名字空了出来，直到 2025 年被另一代云端 Agent 占用——同名是品牌巧合与延续，不是同一产品线的无缝升级。

在 IDE 之外，Codex API 还启发了一批“自然语言驱动操作软件”的实验：生成调用浏览器、设计工具或内部 RPC 的脚本。成功依赖目标软件是否已有良好 API；失败则表现为自信地调用不存在的方法。审查可执行输出，从此成为使用代码模型的基本功，而不是进阶技巧。

含混的人话到了可执行的一侧，必须有人确认动作仍符合原意。
