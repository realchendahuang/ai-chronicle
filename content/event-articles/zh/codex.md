GPT-3 写出一段文字，影响世界仍要经过读者。Codex 写出一段代码，代码本身可以继续调用别的东西。

OpenAI 在 2021 年 8 月发布 Codex API 私有测试版时，抓住的正是这条边界。模型继承 GPT-3，又在公开来源的数十亿行代码上继续训练，包含公开 GitHub 仓库。它最擅长 Python，也覆盖 JavaScript、Go、Perl、PHP、Ruby、Swift、TypeScript 与 Shell 等十多种语言。用于 Python 时，官方称其可利用约 14KB 上下文，而 GPT-3 是 4KB。

发布页面上的演示并不局限于“写一个函数”。自然语言指令被译成代码，代码再操作现成的应用与 API：做游戏、处理数据、控制软件界面。模型无需为每个应用重新学习一种专门的控制协议；只要那个应用已经向程序暴露接口，生成正确代码就可能成为通往它的中间层。

这也解释了 Codex 与普通语言生成的差别。自然语言允许含混，代码必须满足语法、类型、依赖和运行环境；一段看起来合理的程序可能调用不存在的方法，也可能在边界条件下失败。执行能力放大了用途，也放大了审查的必要。OpenAI 以私有测试和免费初期额度逐步开放 API，审核应用并观察开发者怎样使用，而没有把它当成可以放任运行的通用操作者。

在编程内部，Codex 最先触及的是一种常见摩擦：需求已经想清楚，开发者却仍要把它拆成库、函数和调用方式。官方将编程的一部分概括为“把简单问题映射到已有代码”，并认为这是模型最擅长、也最乏味的环节。它还能做代码解释、语言转换和重构，但效果随任务而变。

这个版本的 Codex 后来在 2023 年被弃用；2025 年出现的同名命令行工具和云端软件工程 Agent 属于另一代产品。回看 2021 年，意义不在名字是否延续，而在代码第一次被明确放到语言模型的行动出口上。文字不再只描述要做什么，它可以生成一个可执行的中介。此后的工具调用与编程 Agent 会把这条路走得更远，也会不断重遇同一个问题：谁来确认生成的动作确实是用户想要的动作。
