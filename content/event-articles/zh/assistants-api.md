Assistant、Thread、Message、Run。四个英文名词，在 2023 年 11 月 6 日的 OpenAI 首届 DevDay 上，被写进测试版 Assistants API。

那天之前，许多团队已经会调用模型，却仍各自保存对话、裁剪上下文、处理上传文件、等待工具返回、再把结果拼回提示。Assistants 没有取消这些事，它把它们收进平台状态机：Assistant 存指令与可用工具，Thread 承载一次延续的会话，Message 留下内容，Run 让模型在某个线程上开始处理——必要时检索文件、跑代码解释器，或调用开发者声明的函数。不是“不用写后端”，而是一类重复的 Agent 基础设施，从应用代码挪到了供应商那边。

同一天还有面向终端用户的 GPTs。Assistants 面向要把类似能力嵌进自有产品的人。文件检索托管一部分 RAG，Code Interpreter 托管执行环境，函数仍把业务工具留在开发者一侧。三分法很实用，调试时却容易迷雾：是检索没召回，解释器超时，还是模型选错了函数？责任边界跟着对象走。做一个能读文件、跑代码的助手，实现路径变短；模型调用也不再只是一问一答，而可以在持续存在的线程里推进。平台从出售单次推理，走向承接状态、检索和执行环境。

高度抽象的 API 总会替人做决定。Thread 怎样持久化，Run 进入哪些状态，工具输出以什么顺序回到模型，哪些细节可观察、哪些只能等待返回——便利与限制包在同一层。应用越依赖这些名词，日志、权限与计费仪表盘就越会围着它们生长。约 2025 年 3 月，Responses API 与 Agents SDK 成为新方向，Assistants 进入迁移与弃用路线。改请求地址不够：原先交给 Assistant、Thread 与 Run 的职责——会话持久化、工具循环、文件与代码执行的托管方式——都要在新接口里重新安放。弃用不是对首发的道德判决，而是接口生命周期的后半段。

测试版标签从第一天就在。它鼓励快速搭建，也暗示对象模型仍可能改。四个对象最终没有成为长久标准，却留下一种看待模型应用的方式：一次回答只是最小单位，助手还要经历时间——消息被保存，工具打断流程，运行等待、失败或继续。开发者托付给抽象层的状态没有消失，只是换地方。下一次选择托管深度时，迁移账可以先算清楚：便利曾经到账，账单也会后到。

beta 期间对象模型仍可能调整，却已经足够让大量教程与示例以 Assistant/Thread/Run 为词汇表。词汇表降低沟通成本，也提高迁移成本。当官方指向新运行时，名词折旧变成真实工程。生命周期回望的价值，是承认托管深度会随产品战略摆动——而摆动本身不是道德失败。
