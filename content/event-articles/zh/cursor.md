早期使用网页聊天写代码，有一种反复搬运的疲惫。开发者从编辑器复制函数，切到浏览器解释背景，再把回答粘回来；如果模型缺了另一个文件，还要重新开始。问题不完全是模型不会写，而是它不在工作发生的地方。

Cursor 把这段距离缩短了。它建立在熟悉的 VS Code 基础上，保留扩展生态和编辑习惯，同时把模型聊天、代码库索引与生成式修改放进编辑器内部。文件和符号可以直接成为问题的一部分，语义检索帮助寻找相关代码，修改也不必先经过网页与剪贴板。

2024 年 8 月 22 日，Anysphere 宣布完成 6000 万美元 A 轮融资。这个日期记录的是市场对路线的定价，不是 Cursor 某项能力突然诞生的时刻。到那时，产品已经清楚表达了一种新身份：AI 不再只是传统编辑器里偶尔弹出的补全，而开始参与上下文选择、代码解释和跨位置修改。后来的 Composer 与 Agent 模式继续扩大多文件和终端操作范围，但属于随后演进。

熟悉的外壳降低了迁移门槛，也把竞争推向更细的地方。索引能否及时跟上仓库变化？模型引用了哪些文件？一次批量修改能不能预览和撤销？

这些问题不如基准分数醒目，却直接决定开发者敢不敢接受下一次编辑。模型越主动，撤销、diff 与测试就越需要待在触手可及的位置。

Cursor 带来的顺手是真实的。回答在代码旁出现，修改在仓库里落下，开发者不必不断重新描述上下文。残留的摩擦也同样真实：索引可能过期，相关文件可能漏选，一次看似一致的多文件修改可能把错误扩散得更远。AI 原生编辑器没有取消审查，只是把审查从“要不要粘贴这段代码”扩大为“模型为什么看了这些上下文，并准备同时改这些地方”。

融资新闻很快会被下一轮数字覆盖。Cursor 更持久的变化，是让开发者逐渐无法忍受那段旧距离：代码在这里，模型却在另一个窗口。当生成、检索和编辑被压进同一个工作面，复制粘贴不再是默认动作，编辑器本身也就换了一种中心。
