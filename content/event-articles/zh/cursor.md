Cursor 做的第一个重要选择，是不要求开发者重新学习怎样做一名开发者。

它建立在 VS Code 之上，保留熟悉的扩展、快捷键、文件树和编辑习惯。变化藏在原有动作之间：行内补全不再只猜下几个字符，聊天可以引用文件和符号，代码库经过索引后能够被语义检索，一次指令可以把修改应用到多个位置。用户仍看着代码，仍可以逐处审查；只是从发现问题到形成改动的路径，开始穿过模型。

这条路线与把网页聊天塞进侧栏并不完全相同。侧栏里的模型往往需要人复制代码、解释上下文，再把答案搬回来。Cursor 试图让编辑器本身承担上下文收集：当前文件、选区、相关定义、代码库中的其他位置，都成为交互的一部分。后来的 Composer 和 Agent 模式继续扩大范围，触及多文件修改和终端操作。到 2024 年夏天，Cursor 已经为一个较克制的判断找到不少用户：AI Coding 的竞争也发生在回答抵达代码之前和之后。

8 月 22 日的 A 轮融资公告披露，Anysphere 融资 6000 万美元，并称 Cursor 已有超过四万名客户。公告的语言毫不含蓄，谈论“魔法”、轻盈，甚至有一天写下全世界的软件。这些都是公司的愿景与自我描述，不是经过独立评测的结论。更值得保留的证据在产品结构里：团队投入了下一编辑预测、海量文件检索和快速代码重写系统，说明编辑器的瓶颈不再只是能否调用一个强模型，还包括怎样找到上下文、怎样低延迟地展示修改、怎样让用户保有控制。

Cursor 的流行也改变了“原生”这个词。所谓 AI 原生编辑器，并不一定从空白画布开始，或把文本代码藏起来。它可以借用一个成熟编辑器的外形，在内部重新安排注意力：过去人先在代码库里寻找，再决定写什么；现在模型参与寻找、提出改法，人把更多时间用在判断哪些修改值得留下。

这种变化没有消灭编程中的黏滞。索引会漏掉关系，生成会引入错误，大范围修改越轻松，审查反而越不能省略。可工具史很少由一句彻底替代旧习惯的命令推进。它更常见的方式，是某天以后，人们发现自己已经不愿再把一段代码复制到浏览器里，然后从答案中一行一行搬回来。
