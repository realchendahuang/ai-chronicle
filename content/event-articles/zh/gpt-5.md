GPT-4 系列与 o1 系列曾把两种期待摆在不同入口：一种尽快回答，一种愿意为难题停留更久。模型选择器把差别明白地交给用户，也把一道并不轻松的前置题一并交了出来——写下问题以前，先猜它值不值得等待。

2025 年 8 月 7 日发布的 GPT-5，在 ChatGPT 里首先是一套系统，而不是一个单一固定的推理深度。OpenAI 描述的组合包括一个处理大多数问题的快速模型、一个用于较难任务的深度推理模型，以及判断何时切换的实时路由。路由会参考对话类型、复杂程度、工具需求与用户意图；达到使用限额后，产品还会转入较小的模型。界面看起来统一，背后的计算并不均匀。用户看到的是同一个产品名，系统内部却可能在不同路径上花费不同时间与算力。

若把 GPT-5 只写成“又一个旗舰聊天模型”，会丢掉 2025 年 8 月真正的产品语法。ChatGPT 侧强调路由与限额降级，API 侧强调三档模型与 `reasoning_effort` / `verbosity` 配置；两者共享品牌，不共享控制面。开发者若把聊天里的自动深度推理预期原样搬到 API，会在成本与延迟上误判；普通用户若以为统一入口等于固定智商，会在限额后的降级处感到“突然变笨”而无法解释。机制透明与体验简洁之间的拉锯，是这一代的主线，而不是参数量口号。

这层产品设计很容易与 API 混写，却不能混写。开发者拿到的不是 ChatGPT 那套路由器，而是 GPT-5、GPT-5 mini 与 GPT-5 nano 三档独立模型，可以自行选择成本和延迟，并通过 `reasoning_effort` 调整推理投入、通过 `verbosity` 影响回答展开程度。ChatGPT 试图替普通用户藏起模型调度，API 则把一部分调度权重新交还给应用。把“ChatGPT 里的 GPT-5”与“API 的三档 slug”当成同一对象，会同时弄错开放形态与计费语义。

发布材料把代码、工具调用、数学与专业任务上的成绩放在显眼位置。这些数字来自 OpenAI 在特定设置下的评测，说明厂商希望怎样定义这一代产品，却不是脱离提示、脚手架和评分条件仍然成立的通用保证。更值得留意的是评测单位的变化：一次回答之外，模型还要决定何时推理、能否连续调用工具，以及是否把一项工作做完。性能叙事从“说得多好”部分转向“一整段任务怎样被做完、花了多少”。

自动路由省掉了手动切换，也带来新的不可见部分。用户可能不知道某次回答走了哪条路径，为什么这次等得更久，或限额后的替代模型在什么地方改变了结果。过去写在菜单里的差异，被改写成平台策略；便利与不透明来自同一个动作。应用开发者若依赖 API，则仍要在档位、推理预算和失败重试上做显式选择——产品层隐藏的问题，在工程层重新现身。

GPT-5 没有消灭模型选择，只把它拆成两种产品关系：ChatGPT 尽量代替人分配计算，API 用户仍需自己决定模型档位与推理预算。一个按钮退到幕后，计算并未因此变得简单；把时间再拉长一点，GPT-5 也处在 o 系列之后的产品整合阶段。用户已经习惯“有的模型快、有的模型会想很久”；GPT-5 试图把这种分野收进同一入口，却在 API 上保留档位与推理预算，以免把平台路由强加给所有应用。限额后降级、路由不可见、厂商评测依赖脚手架——这些都不是脚注，而是使用 GPT-5 时必须一并理解的运行机制。若只写“更强的统一模型”，会同时抹掉 ChatGPT 与 API 的分叉，以及快速路径与深度推理路径的并存。

发布材料中的代码与工具调用成绩应标为厂商评测：它们定义了 OpenAI 希望外界如何理解 GPT-5，却依赖提示、工具与评分器设置。更稳妥的阅读方式，是把 GPT-5 当成计算调度产品——在聊天里调度路径，在 API 里调度档位与 effort——而不是当成单一不变的大脑。

GPT-5 没有消灭模型选择，只把它拆成两种产品关系：ChatGPT 尽量代替人分配计算，API 用户仍需自己决定模型档位与推理预算。一个按钮退到幕后，计算并未因此变得简单；决定它怎样分配的人换了。
