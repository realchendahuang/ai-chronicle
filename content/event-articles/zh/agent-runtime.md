在 Agent 成为产品名之前，它常常只是应用里一段越来越长的循环：把问题送给模型，解析它想调用的工具，执行，把结果塞回上下文，再问一次。旁边还要补超时、重试、权限、日志与错误处理。演示里最醒目的是回答；让演示能反复跑的部分，大多没有名字。工程师私下叫它们胶水，发布会很少把胶水请上主台。

2025 年 3 月 11 日，OpenAI 发布 Responses API，并开源 Agents SDK。Responses 统一文本与工具调用响应，内置网页搜索、文件搜索和计算机使用等工具；SDK 把 Agent、Handoff、Guardrail 与 Tracing 变成可组合原语。前者是模型平台上的响应与工具接口，后者是应用侧编排库——配套出现，不是同一层。发布同时给出从 Assistants API 迁移的方向：会话、运行与工具循环曾被四个托管对象收走，现在平台换一套更贴近“一次响应、内置工具、应用编排”的词汇。迁移成本再次提醒：托管深度随产品战略摆动。

内置工具降低接入搜索与文件的门槛，也把权限与计费边界写进平台默认路径。SDK 原语命名不等于系统可靠。Handoff 需要交接什么上下文、Guardrail 拦什么、Tracing 记到哪一层，仍由应用决定。多 Agent 的数量感容易掩盖单点失败：一次错误交接或一次漏记的工具调用，就能让“会协作”的演示在生产里散架。

这一天记下的，是胶水代码终于有了官方名词。名词会再改；工具循环、交接、护栏与追踪这些工作本身，不会因为改名而消失。开发栈考古的价值，是承认：真正让 Agent 产品能反复运行的，往往是那些曾经不好意思写进发布标题的部分。

从 Assistants 到 Responses，名词在变，工作没消失。应用仍要处理失败、权限与可观测性；平台只是提供了另一套默认拼法。Agents SDK 的开源形态让编排逻辑可以进仓库、进代码审查，而不仅是点选配置。胶水有了名字之后，下一问题是：名字能否在生产事故里帮人定位责任。
