在 Agent 成为产品名之前，它常常只是应用里一段越来越长的循环：把问题送给模型，解析它想调用的工具，执行工具，把结果塞回上下文，再问一次。旁边还要补上超时、重试、权限、日志与错误处理。演示里最醒目的是模型回答，真正让演示能够反复运行的部分，大多没有名字。

2025 年 3 月，OpenAI 发布 Responses API 与开源的 Agents SDK，开始为这些中间层命名。Responses API 统一文本输出与工具调用，并直接提供网页搜索、文件搜索和计算机使用等内置工具；Agents SDK 则把 Agent、Handoff、Guardrail 与 Tracing 变成可以组合的原语。前者处理模型如何回应与使用工具，后者处理多个角色和执行过程怎样被应用组织。

“交接”听来像角色扮演，落到运行时却是一项明确决定：当前 Agent 是否应把任务与上下文交给另一个更合适的 Agent。“护栏”也不只是一句安全提示，它要在输入、输出或工具动作周围执行检查。“追踪”更朴素——当系统调用了五次模型、三种工具，又在第二次尝试后得出结果，开发者需要看见这条路径，才知道错误来自模型判断、工具返回还是编排代码。

这些原语不会自动使 Agent 可靠。内置网页搜索仍可能取回不合适的材料，计算机操作仍涉及权限与不可逆动作，Handoff 可能丢失信息，Trace 也只能记录已经发生的失败。平台提供的是一套较一致的施工材料，而不是替应用写好的责任边界。

它改变的，是模型平台竞争的层次。过去开发者主要比较哪一个模型回答更好；当任务开始跨越文件、网页、桌面与多个子角色，接口是否保存推理过程、工具是否容易接入、错误是否能够追溯，也进入选择范围。原本散在框架与业务代码里的胶水，被模型供应商收进正式产品栈。

发布页还给出了从 Assistants API 迁移的方向。这一点并不起眼，却说明 Agent 运行时仍在快速改写自己的抽象：托管多少状态，应用保留多少控制，工具结果以什么形式回到模型，都没有凝固。Responses API 与 Agents SDK 留下的历史痕迹，是行业终于给一个长期藏在应用内部的问题正了名：运行 Agent 本身就是一门需要接口、观测与边界的工程。
